## Overview

The purpose of this feature is to integrate real data fetching for Step 3 (Lập NST đồ) of the Workspace. It restricts image selection in Step 1 to a single image and subsequently fetches the corresponding cropped chromosome images from Firebase Cloud Storage for the selected image to be used in the Karyogram grid.

## Locked Decisions

Decisions extracted during exploring phase:
- **D1 (Single Selection):** Step 1 will strictly limit the user to selecting only 1 metaphase image at a time. Selecting a new image will deselect the previously selected one.
- **D2 (Storage Path & Filtering):** The cropped chromosome images reside in the Firebase Storage path: `test_orders/{orderId}/ai_predict/{selectedImageId}/`. The overview image ending in `_detection.jpg` will be ignored.
- **D3 (Data Source & Metadata):** The Flutter app will prioritize fetching a JSON Manifest generated by the backend (containing URLs and metadata) to avoid slow N+1 `getDownloadURL` calls. If the manifest fails, it will fallback to listing the Storage directory. The chromosome label (slot) will be parsed from the file's Custom Metadata (`current_label`, `original_index`) rather than relying purely on the filename.

## Requirements

### Functional Requirements
- **FR-1:** Modify `WorkspaceCubit` to only allow a maximum of 1 `selectedImageId`. If a user clicks a new image in Step 1, it replaces the currently selected image.
- **FR-2:** Fetch chromosome data by primarily attempting to download a JSON Manifest generated by the FastAPI backend (e.g., `manifest.json` located in the `ai_predict/{selectedImageId}/` folder). This manifest should contain all pre-signed URLs and labels.
- **FR-3 (Fallback):** If the JSON manifest is unavailable, fallback to listing files in the Firebase Storage directory `test_orders/{orderId}/ai_predict/{selectedImageId}/`, ignoring `*_detection.jpg`.
- **FR-4:** Read the chromosome's label (e.g., '11', 'X', 'unassigned') from the Storage file's Custom Metadata (`current_label`). Map this data into local `Chromosome` entities.
- **FR-5:** Update `KaryogramStep` to show a loading state while fetching from Storage/Manifest.
- **FR-6 (Rename Action):** Implement a "Save / Confirm" action in Step 3. This action must identify chromosomes with a new label (slot) and send a request to the FastAPI backend to perform the secure copy/delete operation in Firebase Storage to rename the files.
- **FR-7 (Dirty State Management):** Add an `isDirty` boolean to `WorkspaceState`. Set it to `true` when a user drags and drops a chromosome to a new slot. 
- **FR-8 (Data Loss Prevention):** If `isDirty` is true and the user attempts to change steps or leave the workspace, use `NotificationFactory` to show an `AlertDialog` warning the user about unsaved changes.

### Non-Functional Requirements
- **NFR-1 (Performance):** Use the backend-generated JSON Manifest to avoid the latency of 46 individual `getDownloadURL` API calls on mobile.
- **NFR-2 (Robustness):** Rely on Custom Metadata instead of string parsing for file labels to prevent UI errors caused by unexpected filename formats.

## Acceptance Criteria

- [ ] **AC-1:** In Step 1, clicking an image selects it. Clicking another image deselects the first.
- [ ] **AC-2:** Navigating to Step 3 successfully loads chromosome images using the JSON Manifest (or fallback).
- [ ] **AC-3:** Chromosomes are placed into correct karyogram slots based on the `current_label` found in their Custom Metadata (or Manifest).
- [ ] **AC-4:** Moving a chromosome to a new slot marks the workspace as `isDirty = true`.
- [ ] **AC-5:** Attempting to navigate away while `isDirty = true` triggers a warning dialog via `NotificationFactory`.
- [ ] **AC-6:** Clicking "Save/Confirm" sends a request to the FastAPI backend to rename modified files and resets `isDirty` to `false`.

## Scenarios

### Scenario 1: Fast Loading via Manifest
**Given** the user navigates to Step 3 (since Step 2 is skipped)
**When** the app requests chromosome data
**Then** it downloads `manifest.json` (generated by the backend right after AI analysis in Step 1), parses the 46 URLs and labels instantly, and renders the Karyogram grid without making 46 individual Storage API calls.

### Scenario 2: Unsaved Changes Warning
**Given** the user has moved a chromosome from Slot 11 to Slot 12
**When** the user clicks "Quay lại" (Back)
**Then** an AlertDialog appears warning: "Bạn có thay đổi chưa lưu. Bạn có chắc chắn muốn thoát?"
**And** the navigation is blocked until the user confirms.

### Scenario 3: Saving Layout
**Given** the user has moved a chromosome to Slot 12 and clicks "Save/Confirm"
**When** the save is successful via the backend API
**Then** the `isDirty` flag is reset to `false` and a success notification appears.

## Technical Notes
- **JSON Manifest Generation:** The backend team needs to ensure the AI pipeline generates this `manifest.json` **immediately after the AI analysis in Step 1**, since Step 2 is being skipped.
- **Custom Metadata:** The backend must also attach `current_label` and `original_index` to the Custom Metadata of each `.png` file during the upload process.

## Resolved Questions
- **OQ-1 Resolution:** The file renaming operation (AC-6) will be handled by sending a request to the FastAPI backend. The backend will perform the copy + delete operations in Firebase Storage securely.

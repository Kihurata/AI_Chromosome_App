# Learning: Step 3 Persistence and Step 4 Suggestions

## Patterns

### Firestore as Working Copy
- **What:** Prioritizing Firestore reads over fallback files (like `manifest.json`) ensures user edits are not lost on reload.
- **When to use:** When users are editing data that was originally generated by AI or loaded from a static file.
- **Source:** Recent work on Step 3 karyogram saving.

### Real-time AI Suggestions on Save
- **What:** Returning generated suggestions in the save response reduces unnecessary network requests and provides instant UI feedback.
- **When to use:** When a save action triggers a heavy computation or analysis that the user needs to see immediately.

## Decisions

### Returning Suggestions in Save Response
- **Chose:** Returning suggestions directly in the `save_karyogram` response.
- **Over:** Making the frontend fetch the image document again or poll for changes.
- **Tag:** GOOD_CALL
- **Outcome:** Instant feedback in Step 4 without extra network requests.

### Hardcoding `/api` in Repository URL
- **Chose:** Adding `/api/v1` to the request path.
- **Over:** Trusting the `Dio` base configuration.
- **Tag:** BAD_CALL
- **Outcome:** Caused 404 errors because the base URL already included `/api`.

## Failures

### Removing `material.dart`
- **What went wrong:** Removing the basic material import broke the entire file.
- **Root cause:** Aggressive import cleanup without verifying dependencies.
- **Time lost:** ~5 minutes.
- **Prevention:** Always ensure core Flutter libraries are imported in UI files.

### URL Duplication
- **What went wrong:** Request URL became `http://localhost:8000/api/api/v1/...`.
- **Root cause:** Misunderstanding the `Dio` base configuration in the project.
- **Time lost:** ~10 minutes.
- **Prevention:** Check `injection_container.dart` or existing working requests before modifying URLs.
